## ADR-001：架构模式选择

| 字段 | 内容 |
|------|------|
| **Context** | 单人开发，MVP期日活500-1000，3个月上线。 |
| **Decision** | 采用模块化单体架构（Monolithic），Express后端 + Taro前端 + MySQL。 |
| **Alternatives** | **微服务**（被拒绝：单人团队，领域边界未稳定，运维复杂度高）；**Serverless/CloudBase**（被拒绝：调试困难，与MySQL集成不流畅，成本不确定） |
| **Consequences** | 部署简单，单机可运行；后续拆分微服务需重构；不支持独立扩展某个模块。 |

## ADR-002：前端框架选择

| 字段 | 内容 |
|------|------|
| **Context** | 需同时支持微信小程序和H5网站。 |
| **Decision** | 使用 Taro（React语法）。 |
| **Alternatives** | **原生小程序 + 独立网站**（被拒绝：重复开发，维护成本高）；**uni-app**（被拒绝：React语法生态更成熟，Taro在微信生态优化更好） |
| **Consequences** | 一套代码同时输出小程序和H5；学习成本需掌握Taro API；部分原生能力需插件支持。 |

## ADR-003：后端框架选择

| 字段 | 内容 |
|------|------|
| **Context** | 规则引擎为核心，API调用量不大（日活500-1000，日调用约2000-5000次）。 |
| **Decision** | 使用 Express.js。 |
| **Alternatives** | **Nest.js**（被拒绝：过度工程，单人团队不需要类型约束的复杂度）；**Fastify**（被拒绝：生态不如Express成熟） |
| **Consequences** | 开发速度快，文档丰富；性能足够MVP期需求；后续迁移到其他框架容易。 |

## ADR-004：数据库选择

| 字段 | 内容 |
|------|------|
| **Context** | 核心数据是关系型（节气×地域×体质），食材池数据是半结构化。 |
| **Decision** | 使用 MySQL 8.0。 |
| **Alternatives** | **SQLite**（被拒绝：并发写入能力不足，后续迁移成本高）；**MongoDB**（被拒绝：核心数据是关系型，SQL更自然；食材池用JSON字段存储即可） |
| **Consequences** | 关系型查询效率高；食材池用JSON字段存储，灵活；后续迁移MongoDB容易。 |

## ADR-005：部署方案

| 字段 | 内容 |
|------|------|
| **Context** | 单人运维，成本敏感，需要简单可靠。 |
| **Decision** | 使用腾讯云轻量服务器（2C2G，约¥200/月）。 |
| **Alternatives** | **CloudBase Serverless**（被拒绝：调试困难，与MySQL集成不流畅）；**Docker + Nginx**（被拒绝：运维复杂度高，单人团队不需要） |
| **Consequences** | 部署简单，SSH即可管理；单机故障风险，无高可用；成本可控，¥200/月。 |

## ADR-006：天气API选择

| 字段 | 内容 |
|------|------|
| **Context** | 仅中国大陆用户使用，需要实时天气数据。 |
| **Decision** | 使用和风天气免费版（qweather.com）。 |
| **Alternatives** | **OpenWeather**（被拒绝：国内访问不稳定）；**自建天气爬取**（被拒绝：不稳定，有法律风险） |
| **Consequences** | 免费版每天2000次调用，MVP期足够；数据质量高，国内优化好；后续升级付费版增加调用量。 |

## ADR-007：规则引擎实现方式

| 字段 | 内容 |
|------|------|
| **Context** | 三层参数化推理引擎（CM1不变层→CM2地域适配→CM3体质适配→CM4三维合成）是产品核心。 |
| **Decision** | 使用纯规则引擎（JSON/YAML配置 + 规则脚本），不依赖LLM。 |
| **Alternatives** | **LLM推理**（被拒绝：成本高、延迟大、结果不可控、有幻觉风险）；**静态模板**（被拒绝：无法处理三层参数化推理，菜谱会重复） |
| **Consequences** | 结果可预测、可回溯、可调试；规则变更需要更新配置文件；扩展新节气/城市需要新增配置。 |

## ADR-008：知识数据存储方式

| 字段 | 内容 |
|------|------|
| **Context** | 节气不变层（24条）、地域气候（300条+）、体质适配（9×24条）、天气修正（10条）、食谱池（24×9条）、花草茶池（24×9条）等知识数据需要在运行时快速查询。 |
| **Decision** | 关系型数据存入MySQL，半结构化数据（食材池、花草茶池、天气修正权重）存入JSON字段。 |
| **Alternatives** | **纯YAML/JSON配置**（被拒绝：更新数据需要重启服务，不支持热更新）；**MongoDB**（被拒绝：过度引入，MySQL JSON字段已满足需求） |
| **Consequences** | 查询效率高；JSON字段支持灵活存储；后续迁移MongoDB容易。 |
