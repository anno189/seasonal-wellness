# 顺时养生项目策划书 v1.0

> 日期：2026-07-20
> 状态：草案，待确认
> 核心概念：基于"节气×地域×体质"三层参数化推理引擎的顺时养生产品

---

## 一、项目概述

### 1.1 项目定位

顺时养生是一个**动态决策系统**，不是静态养生内容库。核心差异化：

- 同一个节气，不同城市输出不同方案（地域适配）
- 同一个节气+城市，不同体质输出不同方案（体质适配）
- 每天根据天气自动调整食谱（每日不重复）

### 1.2 核心逻辑

```
节气不变层（方向锚定） → 地域气候适配（程度修正） → 体质适配（侧重点偏移） → 三维养生方案（健康+起居+饮食）
```

### 1.3 产品形态

| 产品 | 定位 | 角色 |
|------|------|------|
| **微信小程序** | 主力产品 | 用户核心交互入口，个性化方案生成 |
| **网站** | 主力产品 | 内容矩阵+SEO引流+品牌形象+用户教育 |

## 二、MVP范围（3个月）

### 2.1 覆盖范围

- **节气**：24节气全量（已有知识蒸馏产物支撑）
- **城市**：3个代表城市（北京/上海/广州），覆盖三大气候带
- **体质**：九种体质全量（已内置九种体质适配表）
- **天气驱动**：每日天气修正（需接入天气API）

### 2.2 核心功能

| 功能 | 优先级 | 说明 |
|------|--------|------|
| 节气养生方案生成 | P0 | 输入节气+地域+体质，输出三维方案 |
| 体质简易测评 | P0 | 8题快速判定体质（是/否选择） |
| 每日天气驱动食谱 | P0 | 每日自动更新，基于当日天气修正 |
| 节气倒计时+推送 | P1 | 推送"明天XX节气，注意..." |
| 节气食材推荐 | P1 | 节气对应的宜食/忌食清单 |
| 网站内容矩阵 | P1 | 节气养生文章/科普/食谱 |
| 用户方案收藏 | P2 | 用户保存和回顾历史方案 |
| 节气日历 | P2 | 全年24节气可视化日历 |

## 三、技术架构

### 3.1 技术栈

- **前端**：Taro（React语法）+ Vant Weapp（小程序UI）/ 自有组件
- **后端**：Node.js（Express/Koa）+ MySQL
- **天气API**：和风天气（免费版足够）或 OpenWeather
- **推送**：微信模板消息（用户主动交互后）
- **内容管理**：简道云/飞书多维表格（轻量CMS）

### 3.2 核心数据表（5张）

| 表名 | 条目数 | 说明 |
|------|--------|------|
| 节气不变层表 | 24条 | 24节气×6维度（天文/气候/阴阳/脏腑/方向/脆弱点） |
| 地域气候表 | 300条+ | N城市×气候类型/温度/湿度/偏差 |
| 体质适配表 | 9×24×N | 九种体质×节气脆弱点映射 |
| 天气修正表 | ~10条 | 天气类型×食材权重修正系数 |
| 食谱模板表 | 24×9×N | 节气+体质→食材清单+模板文本 |

### 3.3 规则引擎逻辑

```
每日菜谱 = 节气方向（不变） + 候阶段（5天递进） + 食材池轮转 + 当日天气修正
```

15天不重复实现路径：
- 候递进规则（初/中/末候，每5天换一次方案强度）
- 食材池轮转（每候从池中随机选择，用日期作为种子）
- 当日天气修正（同一候不同天气，食材池优先级不同）

## 五、商业化策略

### 5.1 收费模式：广告

**选广告的原因**：低门槛，用户免费使用，依靠流量变现。

### 5.2 广告收入模型

| 指标 | 估算 | 说明 |
|------|------|------|
| 广告位 | 首页banner + 方案页插屏 + 文章页信息流 | 3-5个广告位 |
| eCPM | ¥15-30 | 健康类内容中等水平 |
| 日活 | MVP期：500-1000；稳定期：5000+ | 取决于运营 |
| 月收入 | MVP期：¥225-900；稳定期：¥2,250-9,000 | 基于日活×广告位×eCPM×30天 |

### 5.3 后续扩展收入

| 阶段 | 收入方式 | 时机 |
|------|---------|------|
| 第一阶段 | 广告（唯一） | MVP上线后 |
| 第二阶段 | Freemium（个性化深度方案付费） | 用户量5000+ |
| 第三阶段 | 订阅制（全功能+无广告） | 用户量20000+ |
| 第四阶段 | 品牌合作（养生品牌内容合作） | 用户量50000+ |

## 七、运营获客策略

### 7.1 获客渠道

| 渠道 | 策略 | 成本 |
|------|------|------|
| **微信生态** | 节气节点推送+社群裂变+朋友圈分享 | 低 |
| **SEO** | 网站内容矩阵（"春分养生""立秋吃什么"等长尾词） | 低 |

### 7.2 用户留存策略

| 策略 | 说明 |
|------|------|
| **节气倒计时** | 每个节气前推送，维持打开频次 |

## 八、风险评估

| 风险 | 严重度 | 应对策略 |
|------|--------|---------|
| 节气养生内容准确性 | 🔴高 | 所有方案基于《黄帝内经》等经典+中医体质标准，声明非医疗建议 |
| 用户量不足，广告收入低 | 🔴高 | 内容SEO+节气节点集中推广+社群裂变 |
| 食材池数据量不足，菜谱重复 | 🟡中 | 先做3城市×24节气，后续扩展5大区域 |
| 微信推送限制 | 🟡中 | 模板消息仅用户交互后触发，不强制推送 |
| 竞品出现 | 🟢低 | 三层参数化推理引擎是核心壁垒，静态内容库无法复制 |

## 十、关键决策项

以下决策需要在开发前确认：

1. **天气API选择**：和风天气（国内优化好）vs OpenWeather（国际通用）
2. **内容生产方式**：自有撰写 vs AI辅助生成 vs 用户UGC
3. **社区功能**：是否要做用户社区/分享功能（P2，可延后）
4. **小程序命名**：需要确定（影响用户搜索和传播）
5. **免责声明**：需要确认最终文本（PM5诚实边界）
